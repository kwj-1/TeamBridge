## 0. 코드보다 기획이 먼저인 이유

기획 없이 코딩을 시작하면 절반쯤 만든 뒤 "이렇게 하면 안 되겠는데"를 깨닫고 처음부터 다시 짜게 됩니다. 특히 그룹웨어는 **권한(누가 무엇을 할 수 있는가)**과 **결재 흐름**이 복잡해서, 미리 정해두지 않으면 개발 후반에 구조를 갈아엎어야 합니다.

| 기획이 부실하면 | 결과 |
| --- | --- |
| 필요한 기능을 정리 안 함 | 개발 중간에 기능 추가·삭제 반복 |
| 사용자 권한 미정의 | 직원이 남의 결재를 승인하는 등 보안/논리 오류 |
| 화면·URL 미설계 | Controller·경로를 계속 바꾸게 됨 |
| DB 구조 미설계 | 테이블·컬럼을 중간에 고치다 데이터가 꼬임 |

**개발 순서**

```
1단계. 기획   — 무엇을, 누가, 어떤 기능이 필요한지 정리   ← 본 문서
2단계. 설계   — DB(ERD), 화면, URL 구조를 표·그림으로 정리
3단계. 구현   — 설계대로 코드 작성
4단계. 테스트 — 의도대로 동작하는지 확인
```

| 관리자 - 직원 출결 관리 | 전 직원 날짜별 출퇴근 기록 직접 수정 |

> 총 12개 화면. 상세는 `docs/화면설계서.md` 참고.

### 1.4 기술 스택

| 분류 | 기술 |
| --- | --- |
| 백엔드 | Spring Boot, MyBatis |
| 데이터베이스 | MySQL |
| 화면 | Thymeleaf, HTML/CSS |
| 비동기 통신 | fetch API (출퇴근, 결재 승인, 댓글, 캘린더 일정) |
| **실시간 통신** | **WebSocket(STOMP) + SockJS (실시간 채팅)** |
| 외부 API | 날씨(**OpenWeatherMap** 확정) |
| 파일 저장 | 서버 로컬 저장 (프로필 사진, 자료실 첨부) |

> 외부 API·파일 업로드·WebSocket은 필요한 시점에 함께 학습하며 진행.
> 공휴일은 외부 API 연동 없이, **관리자가 캘린더 화면에서 직접 일정을
> 등록**하는 방식으로 처리한다(`CALENDAR_EVENT`, `EVENT_CATEGORY='COMPANY'`).
> 이미 만들어진 "일정 추가" 기능을 그대로 쓰는 것이라 테이블·서식 변경이
> 전혀 없다(2026-07-08 확정, 최초 계획이던 특일정보 API 연동은 폐기).

## 2. 사용자 유형 정의

> 같은 시스템이라도 **사용자에 따라 할 수 있는 일이 달라야** 합니다. 그룹웨어는 특히 "결재를 누가 승인할 수 있는가", "조직 정보를 누가 바꿀 수 있는가"가 핵심입니다.

| 유형 | 정의 | 특징 |
| --- | --- | --- |
| **비로그인** | 로그인하지 않은 상태 | 로그인 화면 외 접근 불가 |
| **일반 직원** | 로그인한 임직원 | 본인 업무 처리(근태·일정·자료·결재 요청·채팅) |
| **팀장** | 팀 결재 1차 승인자 | 직원 권한 + 결재 1차 승인 |
| **부서장** | 부서 최종 승인자 | 직원 권한 + 결재 최종 승인 |
| **관리자(admin)** | 시스템 운영자 | 계정·조직 관리, 공지 작성 |

> **핵심 원칙**: 일반 직원은 본인 것만 처리, 팀장·부서장은 결재 권한 추가, 관리자는 시스템 전반 관리. 부서·직급 등 조직 정보는 직원이 **확인만**, 변경은 관리자만.

### 3.4 실시간 채팅

> WebSocket(STOMP) 기반 실시간 메시지. 공지·결재 같은 "비동기 문서"와 달리 채팅은 "실시간 대화"에 가깝다.

| 기능 | 접근 권한 | 우선순위 | 설명 |
| --- | --- | --- | --- |
| 채팅방 목록 | 본인 | 필수 | 내가 참여한 대화방 목록, 마지막 메시지·시각 |
| 새 대화 시작 (부서→직원 선택) | 직원 | 필수 | 조직도 스타일로 부서 클릭 → 직원 목록 필터 → 선택, 1명 선택 시 1:1방 재사용/생성, 2명 이상 선택 시 그룹방 생성 |
| 1:1 채팅 | 직원 | 필수 | 상대 선택 후 실시간 대화(WebSocket) |
| 실시간 메시지 송수신 | 참여자 | 필수 | 새로고침 없이 즉시 표시(STOMP publish/subscribe) |
| 대화 내역 저장·조회 | 참여자 | 필수 | 방 입장 시 이전 메시지 로드(DB) |
| 안 읽은 메시지 카운트 | 본인 | 필수 | 방별 미읽음 수, 네비 배지 |
| 그룹 채팅 | 직원 | 추가 | 여러 명 참여 방 생성·초대 |
| 온라인 상태 표시 | 직원 | 추가 | 접속 중/오프라인 |
| 채팅 알림 | 본인 | 추가 | 새 메시지 도착 알림 |

> **핵심 기술 포인트**: HTTP 요청/응답이 아니라 **연결 유지형 양방향 통신**. Spring `@EnableWebSocketMessageBroker`, STOMP 엔드포인트, 클라이언트 SockJS/stomp.js 사용. 난이도가 높으므로 1:1 채팅부터 구현하고 그룹은 확장.

### 3.8 전자결재

| 기능 | 접근 권한 | 우선순위 | 설명 |
| --- | --- | --- | --- |
| 결재 작성·상신 | 직원 | 필수 | 유형 선택→양식→결재선·참조자 지정 |
| 받은 결재함 | 승인자 | 필수 | 내가 승인할 문서 |
| 보낸 결재함 | 본인 | 필수 | 내가 올린 문서·진행상황 |
| 참조함 | 참조자 | 필수 | 통보받은 문서 열람 |
| 결재 상세·흐름 | 관련자 | 필수 | 단계별 진행도 시각화 |
| 승인 / 반려 | 팀장·부서장 | 필수 | 순차 승인, 반려 사유 필수 (AJAX) |
| 회수 | 본인 | 추가 | 첫 승인 전 상신 취소 |

**결재 규칙**
- 순차 승인: 앞 단계 승인 후 다음 단계 진행
- 휴가: 팀장까지(1단계) / 비용 등: 팀장 → 부서장(2단계)
- 요청자가 결재선(승인자) 선택, 참조자 다중 지정 가능

**결재 흐름 시각화 예시**
```
[요청자]──▶[① 팀장 ✅승인]──▶[② 부서장 ⏳대기]   참조: 총무팀 김OO
```

**결재 상태**: 임시저장 · 상신 · 진행중 · 승인 · 반려 · 회수

### 3.9 관리자

| 기능 | 접근 권한 | 우선순위 | 설명 |

## 5. 화면 및 URL 설계

```
[인증 / 마이페이지]
GET  /login                 로그인 화면
POST /login                 로그인 처리
GET  /logout                로그아웃
GET  /mypage                마이페이지
POST /mypage/update         내 정보 수정(전화·프로필)
POST /mypage/password       비밀번호 변경

[메인 / 근태]
GET  /main                  메인 대시보드 ("/" → /main)
POST /attendance/checkin    출근 처리 (JSON) ← 로그인
POST /attendance/checkout   퇴근 처리 (JSON) ← 로그인
GET  /attendance/records    본인 월간 출결 기록 조회 (JSON) ?year=&month= ← 로그인

[조직도]
GET  /org                   조직도 (부서 필터 ?deptId=)
GET  /org/member/{id}       직원 상세 (모달/페이지)

[채팅]  ※ 실시간은 WebSocket, 나머지는 HTTP
GET  /chat                  채팅방 목록 화면
GET  /chat/room/{roomId}    채팅방 입장 + 이전 대화 로드
POST /chat/room             채팅방 생성(1:1/그룹) ← 로그인
GET  /chat/room/{roomId}/messages   이전 메시지 조회 (JSON, 페이징)
POST /chat/room/{roomId}/name     그룹 대화방 이름 변경 (JSON) ← 참여자
POST /chat/room/{roomId}/invite   기존 참여자+신규 인원으로 새 그룹방 개설 (JSON) ← 참여자
WS   /ws-stomp              WebSocket(STOMP) 연결 엔드포인트(SockJS)
  SUBSCRIBE /topic/room/{roomId}    방 메시지 구독(수신)
  SEND      /app/chat/{roomId}      메시지 전송(발신)

[공지]
GET  /notice/list           목록(페이징·검색)
GET  /notice/detail/{id}    상세(조회수+1)
GET  /notice/write          작성 폼   ← 관리자
POST /notice/write          작성 처리 ← 관리자
GET  /notice/update/{id}    수정 폼   ← 관리자
POST /notice/update/{id}    수정 처리 ← 관리자
POST /notice/delete/{id}    삭제      ← 관리자

[캘린더]
GET  /calendar              캘린더 화면
GET  /calendar/events       일정 조회 (JSON) ?year=&month=
POST /calendar/event        일정 등록 (JSON) ← 로그인
POST /calendar/event/update/{id}  일정 수정 ← 작성자
POST /calendar/event/delete/{id}  일정 삭제 ← 작성자

[자료실]
GET  /archive/list          목록 ?repoId=  (권한 검증)
GET  /archive/detail/{id}   상세 + 첨부 목록
GET  /archive/write         작성 폼   ← 로그인
POST /archive/write         작성 처리(multipart)
GET  /archive/update/{id}   수정 폼   ← 작성자
POST /archive/update/{id}   수정 처리 ← 작성자
POST /archive/delete/{id}   삭제      ← 작성자/관리자
GET  /archive/download/{fileId}  파일 다운로드(권한 재확인)

[전자결재]
GET  /approval/write        작성 폼(유형 선택)
POST /approval/write        상신 처리
GET  /approval/inbox        받은 결재함(승인 대기)
GET  /approval/outbox       보낸 결재함
GET  /approval/reference    참조함
GET  /approval/detail/{id}  문서 상세 + 결재선
POST /approval/approve/{id} 승인 (JSON) ← 승인자
POST /approval/reject/{id}  반려 (JSON) ← 승인자
POST /approval/cancel/{id}  회수      ← 작성자

[관리자]
GET  /admin                 관리자 메인          ← 관리자
GET  /admin/members         회원 목록            ← 관리자
POST /admin/member/create   계정 생성            ← 관리자
POST /admin/member/reset/{id}    비밀번호 초기화 ← 관리자
POST /admin/member/suspend/{id}  계정 정지       ← 관리자
POST /admin/member/restore/{id}  계정 복구       ← 관리자
POST /admin/member/update/{id}   부서·직급 수정  ← 관리자
GET  /admin/attendance      전 직원 출결 조회 (JSON) ?date= ← 관리자
POST /admin/attendance/{employeeId}  직원 출결 직접 수정 (JSON) ← 관리자
```

> **`{id}` / `@PathVariable`**: 어떤 게시글·문서·직원·방을 다룰지 알려주는 식별자. AJAX 응답(출퇴근·승인·댓글)은 JSON. **채팅은 HTTP가 아니라 WebSocket(STOMP)** 으로, `/app/chat/{roomId}` 로 보내면 서버가 `/topic/room/{roomId}` 구독자에게 실시간 전달.

> 서로 다른 이름(예: 예전 초안의 `chat_member`/`chat_file` vs 확정된

> 않았다 — 결재 대기 알림 등은 별도 테이블 없이 각 화면이 그때그때 조회해서
> 보여주는 방식으로 확정됐다(예: 대시보드의 "결재 대기 건수"는 `APPROVAL_LINE`을

## 8. 개발 우선순위 (제안)

| 순위 | 기능 | 비고 |
| --- | --- | --- |
| 1 | 로그인·회원·권한(인터셉터) | 모든 기능의 전제 |
| 2 | 메인 대시보드 + 출퇴근 |  |
| 3 | 공지 + 관리자(계정관리) | CRUD 기본기 |
| 4 | 조직도 | 부서·직원 데이터 활용 |
| 5 | 자료실(게시판+파일) | 파일 업로드 학습 |

| 8 | **실시간 채팅** | **WebSocket(STOMP) 학습 필요, 1:1 먼저** |
| 9 | 날씨·생일·알림 | 부가 |

> 전자결재는 결재선·상태 흐름을 먼저 확정한 뒤 구현. **실시간 채팅은 WebSocket 개념(연결 유지·publish/subscribe) 학습이 선행**되어야 하므로 후반부에 배치하고 1:1부터 구현.
