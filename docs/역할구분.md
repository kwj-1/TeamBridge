## 0. 시작하기 전에 — 순서 있는 의존관계

4명이 동시에 시작할 수 없는 이유가 있다: **로그인 세션이 없으면 "로그인한
사용자"를 전제로 하는 나머지 모든 화면(마이페이지, 결재, 채팅 등)을 테스트할
방법이 없다.** 그래서 작업은 아래 순서로 진행한다.

```
1단계 (전원 참여, 1~2일)  ── 공통 인프라 킥오프
   - ERD 설계서 5장 SQL(= docs/schema/schema.sql)로 DB 스키마 함께 생성
   - 프로젝트 골격(패키지 구조, application.properties) 합의
   - 김영훈이 LoginInterceptor + 세션 로그인 뼈대를 먼저 완성
        │
        ▼
2단계 (김영훈이 로그인 완료 시점부터, 나머지 3명 병렬 시작)
   - 김영훈: 마이페이지 → 관리자(계정관리) → 관리자(출결관리) 순서로 계속 진행
   - 김우주: 메인 대시보드+출퇴근 → 출결 현황 → 캘린더
   - 정진국: 공지사항 → 자료실 → 전자결재
   - 이다니엘: 조직도 → (WebSocket 학습) → 실시간 채팅
        │
        ▼
3단계 (전원) ── 화면 간 연결 지점 통합 테스트
   - 조직도 "채팅하기" 버튼 → 채팅 1:1방 생성
   - 대시보드의 공지 3건/결재 현황 위젯 응답 형식
   - 전자결재 최종 승인 → 캘린더/근태 반영 트랜잭션 경계
```

**김영훈이 1단계를 최우선으로 끝내야** 나머지 3명이 "로그인된 상태"를 가정하고
개발을 시작할 수 있다. 1단계가 늦어지면 팀 전체 일정이 밀리므로, 1단계는 인터셉터
없이도 임시로 세션에 더미 로그인 사용자를 넣어두는 식으로 빠르게 열어주고,
정식 로그인 폼 연동은 병행해도 된다.

**DB 초기 세팅**: `docs/schema/schema.sql`(전체 17개 테이블) →
`docs/schema/data.sql`(테스트 계정 5개, 전부 비밀번호 "1234") 순서로 각자
로컬 MySQL에 실행하면 동일한 계정으로 바로 테스트 가능.

- [ ] 3. `update/{id}`, `delete/{id}` — 본인 작성 일정만 수정/삭제 가능하도록
  서버에서 작성자 재검증(화면에서 버튼 숨기는 것만으론 부족)
- [ ] 4. 🔗 **협의 필요(정진국)**: 전자결재 최종 승인 시 휴가 일정이
  `ATTENDANCE`(본인 담당 테이블)에 자동 반영되는 부분의 트랜잭션 경계 —
  정진국의 3-3과 같은 서비스 메서드 안에서 두 Mapper를 호출하는 방식 추천

**작업 순서(본인 기준)**: 2-1(출퇴근 AJAX 먼저) → 2-2 → 2-3

## 3. 정진국 — 게시판·문서 도메인

> 프론트 담당(공지사항·자료실·전자결재) 그대로 유지. "단순 CRUD →
> 파일 업로드 → 복잡한 상태흐름" 순서로 난이도가 올라가는 자연스러운 학습
> 곡선이 되므로 이 순서를 그대로 작업 순서로 쓰면 된다. **다만 이 3개 중
> 전자결재가 프로젝트 전체에서 가장 복잡하므로, 본인 파트 중 시간이 가장
> 많이 필요할 것으로 예상된다** — 진행이 늦어지면 자료실을 다른 팀원과
> 나누는 것도 고려(아래 "만약 일정이 빠듯하다면" 참고).

### 3-1. 공지사항 (`notice.js`)
- 테이블: `NOTICE` / API: `GET /notice/list`, `GET /notice/detail/{id}`,
  `GET|POST /notice/write`, `GET|POST /notice/update/{id}`, `POST /notice/delete/{id}`
- 첫 CRUD 화면으로 삼기 좋음(가장 단순) — 다른 게시판형 화면(자료실 등)의 참고 모델

**구현 순서**
- [ ] 1. `NoticeMapper` — 제목/본문 LIKE 검색 + `ORDER BY PINNED DESC,
  NOTICE_ID DESC` 정렬 SELECT(페이징 포함)
- [ ] 2. Controller — `keyword`/`page` 쿼리 파라미터 받아 목록 반환
- [ ] 3. `renderNotices()`를 `fetch('/notice/list?keyword=...')` 호출로
  교체(작성/수정/삭제 버튼 노출은 화면단 판단이어도 되지만, 실제 요청은
  서버가 반드시 재검증)
- [ ] 4. `searchNotices()`를 입력 시마다 1-3의 fetch를 다시 호출하도록 교체
- [ ] 5. 상세 SELECT + `UPDATE NOTICE SET VIEW_COUNT = VIEW_COUNT + 1`
  원자적 증가 쿼리 추가(동시 조회 시에도 정확하도록)
- [ ] 6. `viewNotice()`를 `fetch('/notice/detail/{id}')` 호출로 교체
- [ ] 7. Controller — 작성/수정 각각 권한 검사(`NoticeService.canManageNotice()`:
  `EMPLOYEE_ROLE='ADMIN'` 또는 `POSITION_RANK` 1~2인 팀장/부서장) 후 처리
  (2026-07-15 정진국 결정으로 관리자 전용에서 확장 - 기획서.md v0.9 참고)
- [ ] 8. `NoticeMapper`에 INSERT/UPDATE 쿼리 추가(`NOTICE_ID`는
  `AUTO_INCREMENT`이므로 클라이언트가 만든 id를 받지 않음)
- [ ] 9. `openEditNoticeModal()`을 `GET /notice/update/{id}` 응답으로
  초기값 채우도록 교체
- [ ] 10. `submitNotice()`의 저장 로직을 `POST /notice/write`/
  `POST /notice/update/{id}` 호출로 교체
- [ ] 11. Controller — 7번과 동일한 권한 검사(`NoticeService.canManageNotice()`) 후 삭제
  (소프트 삭제 여부는 팀 논의)
- [ ] 12. `deleteNotice()`의 실제 삭제를
  `fetch('/notice/delete/{id}', { method: 'POST' })`로 교체

### 3-2. 자료실 (`archive.js`)
- 테이블: `REPOSITORY`, `ARCHIVE`, `ARCHIVE_FILE`
- API: `GET /archive/list?repoId=`, `GET /archive/detail/{id}`,
  `GET|POST /archive/write`(multipart), `GET|POST /archive/update/{id}`,
  `POST /archive/delete/{id}`, `GET /archive/download/{fileId}`

**구현 순서**
- [ ] 1. 🔗 **협의 필요(이다니엘, 먼저)**: 파일 저장 경로/파일명 중복 처리
  규칙을 여기서 먼저 확정 — 이 프로젝트에서 파일 업로드를 처음 다루는
  화면이고, 채팅의 `CHAT_ATTACHMENT`(4-3)가 이 규칙을 그대로 따라갈 예정이라

- [ ] 2. `GET /archive/list?repoId=` — 자료실(부서별/전사공용)별 게시글 목록
- [ ] 3. `GET|POST /archive/write` — multipart 업로드, 1번에서 정한 경로 규칙 적용
- [ ] 4. 상세/수정/삭제 — 본인 업로드 파일만 삭제 가능하도록 서버 재검증
- [ ] 5. `GET /archive/download/{fileId}` — 다운로드 시에도 권한 재확인

  접근범위 안 바뀌게)

### 3-3. 전자결재 (`approval.js`) — 가장 복잡
- 테이블: `APPROVAL_FORM_TYPE`, `APPROVAL`, `APPROVAL_LINE`, `APPROVAL_REFERENCE`
- API: `GET|POST /approval/write`, `GET /approval/inbox|outbox|reference`,
  `GET /approval/detail/{id}`, `POST /approval/approve|reject/{id}`,
  `POST /approval/cancel/{id}`

**구현 순서**
- [ ] 1. `APPROVAL_LINE` 1:N 테이블 구조 이해 먼저 — 프론트의
  `signers[]`/`lineStatuses[]` 배열 구조를 그대로 옮기지 말고 테이블 설계에 맞게 재설계
- [ ] 2. `GET|POST /approval/write` — 상신 시 `APPROVAL` + `APPROVAL_LINE`

- [ ] 3. `GET /approval/inbox|outbox|reference`, `detail/{id}` — "현재 단계"는

- [ ] 5. 🔗 **협의 필요(김우주)**: 최종 승인 시 휴가 기간을 `ATTENDANCE`(김우주
  담당 테이블)에 반영하는 부분 — 하나의 트랜잭션으로 묶는 방법을 미리 상의
  (같은 서비스 메서드 안에서 두 Mapper를 호출하는 방식 추천). 연차 일수

**작업 순서(본인 기준)**: 3-1 → 3-2 → 3-3

**만약 일정이 빠듯하다면**: 3-1(공지사항)은 로직이 가장 단순해 다른 팀원이
여유 생겼을 때 도와주기 쉬운 파트다. 반대로 3-3(전자결재)은 도메인 지식이
누적돼야 하므로 중간에 넘기지 않는 걸 권장.

## 4. 이다니엘 — 조직도 + 실시간 채팅

> 프론트 담당은 실시간채팅뿐이었지만, 조직도를 추가로 가져간다. 채팅의
> "새 대화 시작(부서→직원 선택)" 피커가 조직도와 완전히 같은 데이터
> (EMPLOYEE + DEPARTMENT)를 쓰기 때문에, 조직도 조회 로직을 먼저 만들어두면
> 채팅에서 그대로 재사용할 수 있다. 또한 채팅은 WebSocket(STOMP) 학습이
> 선행돼야 해서 상대적으로 가벼운 조직도를 먼저 끝내고 학습 시간을 버는 게
> 좋다.

### 4-1. 조직도 (`org.js`)
- 테이블: `EMPLOYEE`, `DEPARTMENT`, `POSITION`, `ATTENDANCE`
- API: `GET /org?deptId=`, `GET /org/member/{id}`

**구현 순서**
- [ ] 1. `GET /org?deptId=` — 부서별 목록, `WHERE EMPLOYEE_STATUS='ACTIVE'`로
  정지 계정 제외. 직급 서열은 `POSITION_RANK` 컬럼으로 정렬(자바스크립트
  하드코딩 상수는 삭제)
- [ ] 2. `GET /org/member/{id}` — 상세(전화·이메일). **전 직원에게 공개로
  확정**됐으므로 부서별로 가리는 로직은 불필요, 로그인 여부만 확인
- [ ] 3. 🔗 **협의 필요(팀 전체)**: 이 응답 형식을 확정해두면 채팅의
  "새 대화 시작" 피커(4-3), 전자결재의 "참조 대상 선택" 피커(3-3)가 모두
  그대로 재사용 가능 — 나중에 새로 안 만들려면 여기서 필드명을 신경써서 설계

### 4-2. WebSocket(STOMP) 학습 + 설정
- `WebSocketConfig`(`@EnableWebSocketMessageBroker`), STOMP 엔드포인트
  `/ws-stomp`(SockJS), 메시지 브로커 등록
- 참고: `docs/기획서.md` 1.4/3.4, `docs/백엔드_작업_흐름.md` 4-9
- 4-1과 병행 가능 — 조직도 API 개발과 별개로 STOMP 예제 튜토리얼을 미리
  따라해보는 식으로 시간을 확보할 수 있다

### 4-3. 실시간 채팅 (`chat.js`)
- 테이블: `CHAT_ROOM`, `CHAT_ROOM_MEMBER`, `CHAT_MESSAGE`, `CHAT_ATTACHMENT`
- API(HTTP): `GET /chat`, `GET /chat/room/{roomId}`, `POST /chat/room`,
  `GET /chat/room/{roomId}/messages`, `POST /chat/room/{roomId}/name`,
  `POST /chat/room/{roomId}/invite`
- API(WS): 연결 `/ws-stomp`, 구독 `/topic/room/{roomId}`, 발행 `/app/chat/{roomId}`

**구현 순서**
- [ ] 1. HTTP 부분 먼저 — 방 목록(`GET /chat`), 이전 메시지(`GET
  /chat/room/{roomId}/messages`), 방 생성(`POST /chat/room`, 4-1의 조직도
  응답 형식 재사용)
- [ ] 2. WS 연결 — `/ws-stomp` 구독/발행 붙이기(4-2에서 학습한 걸 실제로 적용).
  메시지 송신은 낙관적 렌더링이 아니라 broadcast를 받은 뒤에만 화면에 그리기
- [ ] 3. 🔗 **협의 필요(정진국)**: `CHAT_ATTACHMENT` 파일 저장 규칙을
  3-2(자료실)에서 정한 것과 통일
- [ ] 4. 안읽음 수 계산 — 메시지별 `readBy` 배열 대신 "마지막으로 읽은 메시지"
  커서 방식(`CHAT_ROOM_MEMBER.LAST_READ_MESSAGE_ID`)으로 구현
- [ ] 5. 그룹방 이름 변경(`POST /chat/room/{roomId}/name`), 초대(`POST
  /chat/room/{roomId}/invite`) — 기존 방은 유지하고 새 그룹방을 만드는 지금
  화면 동작 그대로 서버로 이관

**작업 순서(본인 기준)**: 4-1 → 4-2 → 4-3
