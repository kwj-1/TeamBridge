## 1. 지금 프론트가 어떻게 동작하고 있나 (전제)

- 서버가 없고, `js/common.js`의 전역 `state` 객체가 "가짜 DB" 역할을 한다.
- 모든 화면 진입 시 `initPage()` → `loadState()`(localStorage에서 state 복원) →
  `checkAuth()`(로그인 사용자 정보 복원) 순서로 실행된다.
- **로그인 여부와 무관하게 모든 화면에 접근 가능**하다 — 접근 차단(인터셉터)은
  의도적으로 비워뒀고, 이번 백엔드 작업의 1순위 항목이다.
- 데이터가 바뀌는 모든 곳(공지 등록, 메시지 전송 등)은 `saveState()`로
  localStorage 전체를 다시 저장한다. 서버가 생기면 이 전체-저장 방식은 사라지고,
  화면마다 필요한 만큼만 API로 주고받는 구조로 바뀐다.

**백엔드 작업의 핵심은 "이 localStorage 목업 계층을 실제 HTTP/WebSocket 통신으로
1:1 치환하는 것"**이다 — 화면 마크업(html)이나 UI 흐름 자체를 바꿀 필요는 거의 없다.

| 3 | 공지 + 관리자(계정관리) | 공지사항, 관리자 화면 2개 | `notice.js`, `admin.js`, `admin-attendance.js` | CRUD 기본기 습득 |

## 3. 전체 데이터 흐름 (한눈에 보기)

```
[브라우저]                                [Spring Boot 서버]                [MySQL]
  화면(html) ── fetch/폼 제출 ──────────▶ Controller
     ▲                                        │
     │                                        ▼
     │                                    Service (트랜잭션·검증)
     │                                        │
     │                                        ▼
     │                                    Mapper(MyBatis) ───────────▶ 테이블
     │                                        │                          │
     └──────────── JSON/HTML 응답 ◀───────────┴──────────────────────────┘

[예외] 실시간 채팅만 다른 경로를 씀:
  chat.html ── SockJS 연결 ──▶ WS /ws-stomp
     │  발행 /app/chat/{roomId}          @MessageMapping (ChatMessageController)
     │                                        │
     │                                        ▼
     │                                   CHAT_MESSAGE INSERT
     │                                        │
     └── 구독 /topic/room/{roomId} ◀──── 브로커가 구독자 전원에게 broadcast
```

모든 요청은 **LoginInterceptor**(로그인 필요 경로) → 필요 시 **AdminInterceptor**
(관리자 전용 경로: `/admin/**`) → Controller 순서로 걸러진다. 화면에서 버튼을
숨기는 것은 UX일 뿐이고, 실제 차단은 항상 서버 인터셉터/서비스 계층이 한다
(기획서 4장 ⚠️ 참고 — 모든 `js/*.js` TODO 주석에서 반복 강조한 원칙).

## 4. 화면별 요약표

각 항목의 "함수 → API"는 세부 사항이 훨씬 많으므로, 실제 작업 시 해당 `js` 파일
상단의 🔧 주석을 반드시 함께 확인한다. 아래는 색인 용도.

### 4-1. 공통 인프라 (`js/common.js`)
- 필요 테이블: 전체 공통
- 없앨 것: `state.users/notices/...` 전역 배열, `loadState()`/`saveState()`
- 새로 만들 것: `LoginInterceptor`, `AdminInterceptor`, 세션 로그인 사용자 정보
- `getAttendanceRecord()`/`upsertAttendanceRecord()` → 근태 조회/저장 API로 분산

### 4-2. 로그인·마이페이지 (`js/login.js`, `js/mypage.js`)+
- 테이블: `EMPLOYEE`
- API: `POST /login`, `GET /logout`, `GET /mypage`, `POST /mypage/update`, `POST /mypage/password`
- 핵심: 비밀번호 해시 비교, 정지 계정(`EMPLOYEE_STATUS='SUSPENDED'`) 로그인 차단,
  admin 하드코딩(`login.js` 18~20번째 줄) 제거하고 EMPLOYEE_ROLE='ADMIN' 정식 계정으로 전환

### 4-3. 대시보드·출퇴근·출결현황 (`js/dashboard.js`, `js/attendance.js`)
- 테이블: `ATTENDANCE`, `NOTICE`, `APPROVAL`/`APPROVAL_LINE`, `CALENDAR_EVENT`
- API: `GET /main`, `POST /attendance/checkin`, `POST /attendance/checkout`
- ⚠️ `GET /attendance/records?year=&month=` (본인 월간 출결 조회)는 기획서 5장 URL
  목록에 없음 — 신설 필요 (§6 질문 참고)
- 핵심: 지각 판정은 클라이언트 시계가 아니라 **서버 시각 기준**으로

### 4-4. 공지·관리자 (`js/notice.js`, `js/admin.js`, `js/admin-attendance.js`)
- 테이블: `NOTICE`, `EMPLOYEE`, `DEPARTMENT`, `POSITION`, `ATTENDANCE`
- API: `/notice/*` 전체, `/admin/member/*` 전체
- ⚠️ 관리자 출결 관리용 API(`GET/POST /admin/attendance`)도 기획서 5장에 없음 —
  신설 필요 (§6 질문 참고)
- 핵심: `admin.js`의 `positionToRole()` 삭제 — role은 POSITION 조인으로 실시간 판단

### 4-5. 조직도 (`js/org.js`)
- 테이블: `EMPLOYEE`, `DEPARTMENT`, `POSITION`, `ATTENDANCE`
- API: `GET /org?deptId=`, `GET /org/member/{id}`
- 핵심: `POSITION_RANK` 하드코딩 상수 삭제(POSITION 테이블 컬럼으로 대체),
  정지 계정 제외 필터를 서버 쿼리에 반드시 포함

### 4-6. 자료실 (`js/archive.js`)
- 테이블: `REPOSITORY`, `ARCHIVE`, `ARCHIVE_FILE`
- API: `/archive/*` 전체 (작성·수정은 multipart)
- 핵심: `REPO_ID`가 "생성 시점 부서로 고정"되어야 함(작성자 부서 이동 시 접근범위
  안 바뀌게), 다운로드도 서버에서 권한 재확인

### 4-7. 캘린더 (`js/calendar.js`)
- 테이블: `CALENDAR_EVENT`
- API: `/calendar/*` 전체
- 핵심: "6월 고정" 하드코딩(`CAL_WEEKS`) 제거하고 `?year=&month=` 파라미터 기반 동적 생성.
  공휴일은 외부 API 없이 관리자가 직접 등록(`EVENT_CATEGORY='COMPANY'`)하므로
  별도 연동 작업이 없다 — 대신 `POST /calendar/event`가 `COMPANY` 카테고리
  요청을 받으면 관리자 권한인지 서버에서 재검증해야 한다(§6-7 참고).

### 4-8. 전자결재 (`js/approval.js`) — 가장 큰 리팩터링
- 테이블: `APPROVAL_FORM_TYPE`, `APPROVAL`, `APPROVAL_LINE`, `APPROVAL_REFERENCE`
- API: `/approval/*` 전체
- 핵심: `signers[]`/`lineStatuses[]`/`comments[]` 배열 인덱스 구조 → `APPROVAL_LINE`
  1:N 테이블로 전면 재설계. "현재 단계"는 저장하지 않고 조회 시점에
  `MIN(STEP_NO) WHERE LINE_STATUS='WAIT'`로 계산. 최종 승인 시 휴가 기간을
  `ATTENDANCE`에 반영하는 부분은 반드시 트랜잭션으로 묶기

### 4-9. 실시간 채팅 (`js/chat.js`) — 유일하게 WebSocket 사용
- 테이블: `CHAT_ROOM`, `CHAT_ROOM_MEMBER`, `CHAT_MESSAGE`, `CHAT_ATTACHMENT`
- API(HTTP): `/chat`, `/chat/room/{roomId}`, `POST /chat/room`, `/chat/room/{roomId}/messages`
- API(WS): 연결 `/ws-stomp`, 구독 `/topic/room/{roomId}`, 발행 `/app/chat/{roomId}`
- 핵심: 메시지별 `readBy` 배열 방식 → "마지막으로 읽은 메시지" 커서 방식으로 전환,
  메시지 송신은 낙관적 렌더링이 아니라 broadcast를 받은 뒤에만 화면에 그리기
- ⚠️ 그룹방 이름 변경, "초대해서 새 그룹방 개설" 전용 API가 기획서 5장에 없음 —
  신설 여부 논의 필요 (§6 질문 참고)
