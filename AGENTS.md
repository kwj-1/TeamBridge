# TeamBridge — Codex 작업 지침

> ⚠️ 이 파일은 git에 커밋하지 않는다(비공개 기획 문서와 같은 폴더에 보관).
> 팀원 각자 이 파일을 **본인 TeamBridge 저장소 루트**(`TeamBridge/AGENTS.md`)에
> 수동으로 복사해 넣어야 Codex가 세션 시작 시 자동으로 읽는다.
> 내용이 갱신되면 팀장이 다시 배포하니, 그때마다 본인 사본을 덮어쓸 것.

## 0. 이 문서보다 먼저 볼 것

- 처음 Codex를 쓰는 팀원은 같은 폴더의 `AI_협업_지침서.md`부터 읽을 것
  (AI에게 뭘 맡겨도 되는지, git push는 언제/어떻게 하는지 초보자용 설명)

## 1. 프로젝트 개요

- 사내 그룹웨어: Spring Boot + MyBatis + Thymeleaf + MySQL
- 공식 팀 저장소: `TeamBridge0701/TeamBridge` (git remote 이름은 보통 `team`)
- 각자 개인 fork에서 `work` 브랜치로 작업 후, 팀 저장소 `main`으로 PR
- **기획 문서(역할구분.md, 기획서.md, ERD_설계서.md, 화면설계서.md,
  백엔드_작업_흐름.md)는 비공개 정보라 git 저장소에 올리지 않는다.**
  팀장에게 별도로 전달받은 사본을 로컬에 보관해서 참고할 것 — Codex에게
  물어볼 때도 "역할구분.md 파일 경로는 OO야, 참고해서 답해줘"처럼 본인
  PC의 실제 경로를 알려줘야 Codex가 읽을 수 있다.

## 2. 담당 영역 — 본인 파트만 수정

(4인 역할 분담 요약 — 상세 체크리스트·구현 순서는 별도 전달받은
`역할구분.md` 참고)

| 담당자 | 도메인 |
| --- | --- |
| 김영훈 (팀장) | 공통 인프라, 로그인, 마이페이지, 관리자(계정/출결) |
| 김우주 (부팀장) | 대시보드+출퇴근, 출결 현황, 캘린더 |
| 정진국 | 공지사항, 자료실, 전자결재 |
| 이다니엘 | 조직도, 실시간 채팅 |

- 다른 사람 담당 파일을 고쳐야 할 것 같으면, **조용히 고치지 말고** 먼저
  사용자에게 "이건 OO 담당 영역인데 수정해도 되냐"고 확인할 것.
- 다음은 여러 사람이 공유하는 부분이라 **임의 변경 금지, 반드시 협의 후 진행**:
  - 공통 인프라: `LoginInterceptor`, `AdminInterceptor`, `WebConfig`, `SecurityConfig`
  - DB 스키마(`schema/schema.sql`, `schema/data.sql`) — 컬럼/테이블 추가·변경
  - `ATTENDANCE` 테이블의 UNIQUE 제약, 지각 판정 규칙 (김영훈 ↔ 김우주)
  - 조직도 `GET /org` 응답 형식 (전원이 재사용)
  - 파일 업로드 저장 경로/파일명 규칙 (자료실 ↔ 채팅)
  - 그 외 협의가 필요한 항목은 `역할구분.md`의 🔗 표시와 6장 색인 참고

## 3. 코딩 규칙

- 이미 팀이 쓰고 있는 패턴을 따를 것: MyBatis Mapper+XML, 컨트롤러에서
  `HttpSession`으로 로그인 사용자 조회, 필드가 적은 요청은 DTO 대신
  `@RequestParam` 사용 등. 새 패턴을 임의로 도입하지 말 것.
- 요청받은 범위 이상으로 리팩터링하거나 추상화를 추가하지 않는다.
- 주석은 "왜"가 코드만 봐서는 드러나지 않을 때만 한 줄로.
- 서버에서 재검증 없이 화면단 권한 체크만으로 끝내지 않는다 (예: 버튼 숨김만으로
  삭제 권한을 막지 않기).

## 4. 진행 방식 — 한 번에 다 짜지 않기

> 팀원 대부분이 AI로 코드를 짜본 적이 없다. 이해 못 한 코드가 쌓이면 나중에
> 버그가 나도 원인을 못 찾으므로, Codex는 아래 순서를 기본값으로 한다.

- 사용자가 "전체 다 만들어줘", "한 번에 끝내줘"처럼 요청해도, 곧바로 전체
  구현을 쏟아내지 않는다. `역할구분.md` 체크리스트의 항목 단위(또는 그만한
  크기)로 하나씩 끊어서 진행한다.
- 각 단계가 끝나면: (1) 방금 만든 코드가 무엇을 왜 이렇게 짰는지 짧게 설명하고,
  (2) 사용자가 이해했는지 확인한 뒤에만 다음 단계로 넘어간다. "이해됐어?"
  "여기까지 실행해보고 문제없으면 다음 단계로 갈까?" 정도로 물어보면 충분하다.
- 아래는 특히 멈춰서 설명·확인을 받아야 하는 지점이다 — 대충 넘기면 사고로
  이어지기 쉽다:
  - 로그인/세션/권한 체크 (`LoginInterceptor`, `AdminInterceptor`, `PasswordEncoder`)
  - 여러 사람이 공유하는 테이블/트랜잭션 (`ATTENDANCE` 지각 판정, 전자결재→근태
    반영 트랜잭션 경계 등 2장의 협의 항목)
  - SQL Injection·XSS 등 보안과 직결되는 부분(파라미터 바인딩, 서버 재검증)
- 이건 강제 잠금 장치가 아니라 기본 동작 원칙이다. 사용자가 "이해했으니 이 부분은
  건너뛰고 다음 걸로 넘어가줘"라고 명시적으로 요청하면 그대로 따른다 — 목표는
  이해 없이 넘어가는 걸 "기본값"으로 만들지 않는 것이지, 사용자를 막는 게 아니다.

## 5. Git / 커밋 / 푸시 규칙 (중요)

- **pull/push/merge 등 git 명령을 실행해달라는 요청을 받으면, 곧바로 실행하지
  않고 먼저 어떤 명령을 왜 그 순서로 실행하는지 설명한다** (예: "지금은
  `git fetch team` → `git merge team/main` 순서로 팀 최신 내용을 받아와야 해,
  왜냐하면..."). 설명을 들은 뒤에도 사용자가 "그냥 해줘"라고 하면 그때 실행한다.
  목적은 대신 해주는 걸 막는 게 아니라, 매번 순서를 몰라서 헤매지 않도록 하는 것.
- **커밋 전에는 항상 diff를 보여주고 사용자 확인을 받는다.** 사용자가 명시적으로
  요청하지 않으면 Codex가 먼저 커밋하지 않는다.
- `git push`, `git push --force`, `git reset --hard`, `git branch -D` 등은
  사용자가 명시적으로 요청한 경우에만 실행한다.
- 커밋은 본인 fork의 `work` 브랜치에서만 진행한다. 팀 저장소(`team` remote)의
  `main`에는 직접 push하지 않는다 — 반드시 PR을 통해서만 병합한다.
- 작업 시작 전에는 `team/main`을 최신으로 받아온 뒤 시작한다(오래된 브랜치에서
  작업하면 나중에 충돌이 커진다).
- 커밋 메시지는 "무엇을 왜 바꿨는지" 한 줄로 — 이미 리포에 쓰인 스타일
  (`feat:`, `fix:`, `refactor:` 접두어 + 한글 설명)을 따른다.
- **이 AGENTS.md, 기획 문서, `docs/` 폴더 자체를 절대 `git add`하지 않는다** —
  비공개 문서가 실수로 커밋되는 걸 막기 위해, `git add -A`/`git add .` 대신
  변경한 파일만 콕 집어서 add할 것.

## 6. 보안/민감정보

- `application.properties`의 DB 계정은 로컬 개발용 값(root/1234, localhost)만
  사용한다. 실제 운영 자격증명이나 개인정보가 담긴 값은 커밋하지 않는다.
- 업로드 경로처럼 개인 PC 전용 경로가 하드코딩돼 있다면, 공용으로 올리기 전에
  팀과 확인한다.
- 기획 문서·역할 분담표 등 비공개 문서는 TeamBridge 저장소 폴더 안에 두더라도
  git이 추적하지 않는 위치(예: 저장소 바깥, 또는 `.gitignore` 처리된 폴더)에만
  보관한다.
