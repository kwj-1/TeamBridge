# AI(Claude Code) 협업 지침서

> 대상: TeamBridge 팀원 전원, 특히 AI로 코드를 짜본 적이 없는 팀원
> 목적: Claude Code를 각자 써도 결과물이 어긋나지 않게, git push 습관을 통일하기 위함
> ⚠️ 이 문서와 같은 폴더의 기획 문서(역할구분.md 등)는 비공개라 git에 올리지 않는다.
> 팀장에게 전달받은 사본을 각자 로컬에 보관해서 참고할 것.

---

## 1. Claude Code 쓰기 전에 꼭 알아야 할 것

### 1-1. `CLAUDE.md`를 본인 TeamBridge 폴더 루트에 먼저 넣어둔다
같은 폴더에 있는 `CLAUDE.md`를 복사해서 본인 `TeamBridge/CLAUDE.md`에 넣는다
(이 파일은 git에 올라가지 않으니 각자 수동으로 넣어야 한다). 넣어두면
Claude Code가 해당 폴더에서 세션을 시작할 때마다 자동으로 읽어서, 담당 영역·
코딩 규칙·git 규칙을 매번 다시 설명하지 않아도 된다. Claude가 이상하게
행동하면 "CLAUDE.md 있어? 읽었어?"라고 물어보면 된다.

### 1-2. Claude가 만든 코드는 반드시 이해하고 넘어간다
- 무슨 코드인지 모르겠으면 "이 코드 뭐 하는 거야?"라고 바로 물어볼 것.
  이해 안 된 채로 다음 단계로 넘어가면, 나중에 버그가 나도 원인을 못 찾는다.
- 특히 SQL(Mapper XML)과 세션/권한 체크 부분은 반드시 이해하고 넘어갈 것 —
  이 프로젝트는 로그인한 사용자 기준으로 데이터가 갈리기 때문에 대충 넘기면
  다른 사람 정보가 보이는 사고로 이어질 수 있다.

### 1-3. 본인 담당 영역만 작업 범위로 준다
- 전달받은 `역할구분.md`에 4명의 담당 모듈이 정리돼 있다. Claude에게 일을
  시킬 때 "내 파트인 OO 화면만 고쳐줘"처럼 범위를 명확히 준다.
- 다른 사람 파일까지 고치려는 제안이 나오면 (예: 공통 인터셉터, DB 스키마)
  먼저 팀에게 물어보고 진행한다. 혼자 고치고 push했다가 다른 사람 작업과
  충돌하는 게 가장 흔한 사고 유형이다.

### 1-4. 권한 확인 프롬프트가 뜨는 이유
Claude Code가 파일을 고치거나 명령어(git, 터미널)를 실행하기 전에 "이 작업을
허용할까요?" 라고 물어보는 창이 뜬다. 이건 실수 방지 장치이니 **내용을 읽고
이해한 다음 승인**한다. 특히 아래 명령은 뭔지 모르면 거절하고 팀장/Claude에게
"이게 뭐 하는 명령이야?"라고 먼저 물어본다.

- `git push --force` (강제 푸시 — 다른 사람 커밋을 지울 수 있음)
- `git reset --hard` (내 로컬 변경사항을 되돌릴 수 없이 삭제)
- `git checkout -- .` / `git clean -f` (커밋 안 한 변경사항 삭제)
- `git branch -D` (브랜치 강제 삭제)
- `git add -A` / `git add .` (비공개 문서·CLAUDE.md까지 실수로 같이 올라갈 수 있음 —
  항상 바뀐 파일만 콕 집어서 add)

### 1-5. 처음 보는 개념은 코드보다 설명부터
각자 파트에 "이번 프로젝트에서 처음 다뤄보는 개념"이 하나씩 있다 — 예를 들어
정진국의 결재선(`APPROVAL_LINE` 1:N 구조)·파일 업로드, 이다니엘의
WebSocket/STOMP, 김우주의 트랜잭션 경계 등. 이런 부분은 Claude에게 바로
"코드 짜줘"라고 하지 말고, 먼저 "이 개념이 뭔지 설명해줘"부터 물어보고
이해한 다음 코드를 요청한다. 개념을 모른 채로 받은 코드는 나중에 비슷한
문제가 또 나왔을 때 응용이 안 된다.

## 2. Git 작업 순서 (fork + work 브랜치)

이 팀은 각자 GitHub에 **본인 fork**를 만들고, 그 안에서 `work` 브랜치로
작업한 뒤, 팀 저장소(`TeamBridge0701/TeamBridge`)의 `main`으로 PR을 보내는
방식을 쓴다. 아래 순서를 매번 그대로 따른다.

### 2-1. 작업 시작 전 (하루 시작할 때마다)
```bash
git checkout work
git fetch team          # 팀 저장소의 최신 내용 받아오기
git merge team/main      # 내 work 브랜치에 팀 최신 내용 합치기
```
- `team`은 팀 공식 저장소를 가리키는 remote 이름이다. 안 잡혀 있으면
  `git remote add team https://github.com/TeamBridge0701/TeamBridge.git`로 추가한다.
- 이 단계를 건너뛰고 며칠씩 묵혀서 작업하면, 나중에 합칠 때 충돌이 커진다.
  **최소 하루 한 번은 받아올 것.**

### 2-2. 작업 중 — 작게, 자주 커밋
- 화면 하나, 기능 하나가 끝날 때마다 커밋한다. 하루치 작업을 커밋 1개로
  몰아넣지 않는다 — 문제가 생겼을 때 어디서부터 잘못됐는지 찾기 어려워진다.
- 커밋 메시지는 "무엇을 했는지"를 짧게: `feat: 공지사항 목록 조회 API 추가`,
  `fix: 출근 버튼 중복 클릭 방지` 처럼.
- 커밋하기 전에 `git status`로 뭐가 올라가는지 한 번 확인한다 — CLAUDE.md나
  비공개 문서가 실수로 껴 있지 않은지 체크.

### 2-3. 작업 마무리 — push
```bash
git push origin work     # origin = 내 개인 fork
```
- **최소 하루 1회는 push한다.** 로컬에만 며칠 쌓아두면 컴퓨터 문제 생겼을 때
  작업이 통째로 날아갈 수 있고, 팀원들이 진행 상황을 못 본다.
- push 대상은 항상 **본인 fork**(`origin`)다. 팀 저장소(`team`)에는 직접
  push하지 않는다.

### 2-4. PR(Pull Request) 만들기
- 화면/기능 단위로 어느 정도 완성되면, 본인 fork의 `work` →
  `TeamBridge0701/TeamBridge`의 `main`으로 PR을 만든다.
- PR 설명에 "뭘 했는지"만 간단히 적어도 된다. 리뷰는 팀장이 병합 전 확인한다.
- **DB 스키마를 바꿨다면 PR 설명에 반드시 표시** — 다른 사람도 로컬 DB에
  같은 변경을 반영해야 한다.

### 2-5. 충돌(conflict)이 났을 때
- 당황하지 말고 어떤 파일에서 충돌났는지부터 확인한다 (`git status`).
- **공용 파일**(인터셉터, 스키마, 공통 JS)에서 충돌났다면 혼자 임의로
  결정하지 말고, 관련 담당자에게 먼저 물어본다.
- 본인만 건드린 파일의 충돌이면 Claude에게 "이 충돌 내용 설명해줘"라고
  물어보고, 어느 쪽 코드를 남길지 이해한 다음 해결한다.

## 3. 하루 체크인 (오전/오후 미팅에 얹기)

이미 하고 있는 오전/오후 미팅에서, 각자 아래 3줄만 말로 공유한다. 따로
기록하거나 문서화하지 않아도 된다 — 목적은 🔗 협의가 필요한 부분이 실제로
부딪히기 전에 말로 미리 걸러내는 것.

1. 오늘/오전에 한 것
2. 막힌 것
3. 협의 필요한 것 (있으면 — 아래 4장 표에 있는 항목인지 바로 대조)

"협의 필요한 것"이 나오면 그 자리에서 상대방과 바로 맞추고 넘어간다 — 나중에
각자 PR을 올리고 나서 충돌로 발견하면 되돌리는 비용이 훨씬 크다.

## 4. 협의가 필요한 지점 (🔗)

전달받은 `역할구분.md` 6장에 담당자 간 협의가 필요한 항목이 표로 정리돼
있다. 아래 항목은 **작업 전에 먼저 상대방과 맞추고 시작**할 것 — 나중에
합치면 더 오래 걸린다.

| 협의 대상 | 내용 |
| --- | --- |
| 김영훈 ↔ 김우주 | `ATTENDANCE` UNIQUE 제약, 지각 판정(09:00 초과) 규칙 |
| 김우주 ↔ 정진국 | 대시보드 공지/결재 요약 응답 형식, 휴가 승인→ATTENDANCE 반영 트랜잭션 |
| 정진국 ↔ 이다니엘 | 파일 업로드 저장 경로·파일명 중복 처리 규칙 |
| 이다니엘 ↔ 전원 | 조직도(`GET /org`) 응답 형식 |
| 전원 | 사번 자동 채번 규칙 |

## 5. 하지 말아야 할 것 (요약)

- Claude가 제안한 커밋을 이해 안 된 채로 그냥 승인하지 않는다.
- 팀 저장소(`team`)의 `main`에 직접 push하지 않는다.
- 다른 사람 담당 화면/테이블을 미리 상의 없이 고치지 않는다.
- `git push --force`, `git reset --hard` 같은 명령을 뭔지 모르고 승인하지 않는다.
- 며칠씩 커밋/push 없이 로컬에만 쌓아두지 않는다.
- `git add -A`/`git add .`로 통째로 add해서 비공개 문서(CLAUDE.md, 기획 문서)를
  실수로 커밋하지 않는다.
